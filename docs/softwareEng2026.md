# ソフトウェア工学のまとめ
前半では、ソフトウェア工学概論とプロジェクトマネジメント、WBSについてを学んだ。\
まずソフトウェア工学の難しいところは、実務から体系化された知見が必要であること、知見を理解しても実践が難しいこと、個々人のソフトウェア工学の知識だけでは不十分であり、チーム全体で共有する必要があることの3つである。\
次はソフトウェアについてである。\
ソフトウェアの特徴はいくつかあるが、共通しているのは適応や強化、拡張や再構築といった変化が必要なことである。具体的には環境やニーズ、システムに合わせた変化が必要である。また、ソフトウェアはハードウェアと異なり劣化しないが、悪化はする。\
次はソフトウェア工学についてである。\
ソフトウェア工学は「品質」、「コスト」、「納期」の最適なバランスを実現するための手法・方法論である。\
ソフトウェア工学の必要性は４つあり、ソフトウェアの大規模化と複雑化、IT人材の不足、不測の事態への対応、不確実性の増大が挙げられる。これらを解決するためにソフトウェア工学を学ぶ必要がある。\
次はソフトウェアライフサイクルについてである。\
ソフトウェアライフサイクルはソフトウェアの誕生、開発・運用、そして廃止という流れとなっている。\
具体的にはまず計画を行い、それに従って要件定義をする。要件定義の意義はプロジェクト関係者のバイブルを制作すること、社内外への宣言と協力依頼、RFPが挙げられる。\
その次は設計を行う。これは要件定義書を元として設計書を作成する作業であり、コーディングが出来るレベルの仕様書に落とし込む。\
その後制作、テストとデバッグを行い、稼働後は運用・保守を続ける。運用・保守の際には開発者がプロジェクトから離れている可能性も考慮する必要がある。\
次はソフトウェア分析についてである。\
ソフトウェア分析はソフトウェアを見える化するための手法であり、物量、実行性能、開発工数、品質に関する分析をそれぞれ行う。ソフトウェアには優先すべき要素が品質、コスト、納期の3つ存在しているが、ソフトウェアの種類によって優先度は異なる。例えば人命に影響するシステムは品質を、公共関連システムはコストを、イベント行事関連システムは納期を優先する。\
次は開発プロセスについてである。\
開発プロセスは多く存在しているが、一長一短であり場面や要求によって適した手法が変わる。品質重視なら計画駆動型、納期重視なら反復型、コスト重視なら反復型、アジャイル型が適している。\
最後はWBSについてである。\
WBSとはプロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、成果物を主体に階層的に要素分解したものである。\
WBSのメリットとしてはスコープが明確になること、やるべきことが明確になること、全体管理と作業計画が明確化されること、プロジェクト実施時はWBSに則り実行するのみとなることが挙げられる。\
後半ではコーディングやgithub、CI/CDについて学んだ。\
コーディングは読まれることの方が多いため、読みやすいコードが必須である。そのために1行の長さやスペース、改行などについてルールが定められている。そのルールを満たしているかチェックするflake8やpylintというツールがある。\
githubはリモートレポジトリのホスティングサービスの一つで、オープンソースソフトウェアの主要なポータルサイトである。githubではレポジトリに関する問題・課題・バグ・機能追加や質問などを挙げることができ、それらを管理することもできる。\
githubにはブランチの作成、切り替え機能もあり、安全にコードを書き換えることが可能である。\
次はCI/CDについてである。\
CIはコード変更を共有リポジトリに頻繁に統合するプロセスで、自動テストとビルドを定期的に実行し、バグ早期発見と修正を可能にしている。CIの自動化により開発のスムーズな進行が促進される。\
CDはコード変更をテスト環境や本番環境に自動的にデプロイするプロセスで、自動デプロイメントを組み込み、手動操作によるデプロイ必要性を排除する。CDにより、ユーザーフィードバックを迅速に反映可能となる。\
CIの例として、flake8を適用でき、pythonのコードを自動でチェックすることが可能となる。
